# Technical Plan — SPEC-002: GitNexus Deep Integration

## 1. Approach

This is a **configuration + documentation** spec — no application code changes. The work consists of:

1. **One config change**: Add `.gitnexus/` to `.gitignore`
2. **Five command enhancements**: Add/improve GitNexus MCP tool references in pipeline commands
3. **Two runtime commands**: `npx gitnexus analyze --skills` + `npx gitnexus setup` (executed during implementation, not committed)

All command modifications follow the same pattern already established in `primitiv.plan.md`: GitNexus tools as primary with specific tool names, graceful manual fallback, document findings.

### Pattern to Replicate

From `primitiv.plan.md` (lines 27-36), the established pattern is:

```markdown
1. **Use GitNexus MCP tools** (if available):
   - `gitnexus.query` — <specific purpose for this step>
   - `gitnexus.context` — <specific purpose for this step>

2. **Fallback: Manual search** (if GitNexus not available):
   - <equivalent manual steps using Glob/Grep/Read>
```

All five commands will follow this same structure.

### 3.2 `.claude/commands/primitiv.plan.md` — Enhance existing references

**Current state:** Step 2 lists `query`, `context`, `impact` generically.

**Enhancement:** Add per-sub-step specificity to the existing Step 2:
- When searching for related implementations → `query` with relevant keywords
- When understanding a specific symbol/function → `context` with the symbol name
- When assessing blast radius of planned changes → `impact` with the entry point
- Add a note about using `gitnexus://repo/{name}/clusters` resource to understand functional areas

Keep the existing fallback section unchanged.

### 3.3 `.claude/commands/primitiv.tasks.md` — Add GitNexus section

**Current state:** No GitNexus references. Step 2 says "Generate tasks" without codebase analysis.

**Enhancement:** Insert a new step between "Load context" and "Generate tasks":

> **Step 1.5: Codebase Analysis for Task Dependencies**
>
> Use GitNexus MCP tools (if available):
> - `gitnexus.impact` — For each planned file change, assess blast radius to identify dependent files that must be included in the same task
> - `gitnexus.context` — Get 360-degree view of key symbols to understand which files are tightly coupled and should be grouped together
>
> Fallback: Manual search using Glob/Grep to find imports and references.
>
> Use findings to: group tightly-coupled files into the same task, order tasks by dependency depth, and list affected files per task.

### 3.4 `.claude/commands/primitiv.implement.md` — Enhance existing reference

**Current state:** Step 2b mentions `gitnexus.context` to understand files before modifying.

**Enhancement:** Expand into a structured section:
- **Before modifying each file**: Use `gitnexus.context` to understand the symbol being changed + `gitnexus.impact` to check blast radius before making changes
- **After completing all tasks**: Use `gitnexus.detect_changes` to verify that all impacted files have been addressed and no unintended side effects remain
- Add fallback: manual search with Glob/Grep

### 3.5 `.claude/commands/primitiv.test-feature.md` — Add GitNexus section

**Current state:** No GitNexus references. Step 2 determines test types needed.

**Enhancement:** Insert codebase exploration step before test generation:

> **Step 1.5: Discover Test Boundaries (GitNexus)**
>
> Use GitNexus MCP tools (if available):
> - `gitnexus.query` — Search for existing tests related to the spec's domain to avoid duplicates and follow existing patterns
> - `gitnexus.context` — For each function under test, discover all callers/callees to understand integration boundaries
> - `gitnexus.impact` — Identify all code paths affected by the spec's changes to ensure complete test coverage
>
> Fallback: Manual search using Glob/Grep to find related test files and imports.
>
> Use findings to: ensure tests cover all affected code paths, follow existing test patterns, and avoid duplicate test coverage.

### 3.6 `.claude/commands/primitiv.compushpr.md` — Add GitNexus section

**Current state:** No GitNexus references. Step 5 creates PR body from spec/plan/tasks.

**Enhancement:** Add a step before PR creation:

> **Step 4.5: Impact Analysis for PR Description**
>
> Use GitNexus MCP tools (if available):
> - `gitnexus.detect_changes` — Analyze the git diff to map all affected processes, symbols, and downstream dependencies
>
> Fallback: Use `git diff --stat` for a basic file-level change summary.
>
> Include the impact analysis in the PR body under a `## Impact Analysis` section, listing affected functional areas and downstream dependencies.

### 3.7 Runtime: Initial GitNexus Setup (not committed)

During implementation, execute:

```bash
npx gitnexus analyze --skills   # Index repo + generate community skills
npx gitnexus setup              # Auto-install skills, hooks, validate MCP
```

These commands generate:
- `.gitnexus/` directory (gitignored)
- `.claude/skills/` with core + community-specific skills (generated, not hand-written)
- PostToolUse hooks for auto-reindexing

## 4. Architecture

No architectural changes. All modifications are to markdown command files (developer instructions) and one config file (`.gitignore`). The GitNexus knowledge graph operates as an external dev-time tool accessed through the existing MCP server — no new system boundaries, data flows, or dependencies are introduced into the application.

## 5. Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `npx gitnexus analyze` fails on this codebase | Low | The codebase is standard TypeScript — well within GitNexus's supported languages. If it fails, the graceful fallback pattern means no commands break. |
| `npx gitnexus setup` overwrites `.mcp.json` | Medium | Verify after setup that `.mcp.json` still matches the current config. If overwritten, restore from git. |
| Community-specific skills are low quality (small codebase) | Medium | The core 4 skills are always useful regardless. Community skills are bonus value — if empty, no harm done. |
| PostToolUse hooks slow down commits | Low | GitNexus reindexing is incremental and fast. Monitor first few commits for latency. |
| Future GitNexus updates break MCP tool names | Low | Tool names (`query`, `context`, `impact`, `detect_changes`) are stable public API. Pin `@latest` for now. |
